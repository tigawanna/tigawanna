## Source Layout (`src/`) — TanStack Start / TanStack Router only

> **Stack scope:** This section applies **only** to TanStack Start and TanStack Router web apps. It does **not** apply to **Expo** / Expo Router apps, Electron shells, or other stacks.
>
> **This repo:** `apps/site/src/`

### Tree map

### `routes/`

TanStack Router turns any `*.tsx` route file that exports a route definition into a URL. To **opt out** of routing for a folder, prefix with `-`:

| Prefix | Use for |
|--------|---------|
| `-components/` | UI only used on this route (or route group). Copy-paste friendly bundles (e.g. a sign-in screen tied to one flow). |
| `-utils/` | Helpers only needed on this route. Avoids cluttering global `utils/`. |
| `-hooks/` | Hooks that encapsulate local UI/state for this route. Not preferred for data fetching — use `queryOptions` in components instead. |

**Hoist upward when reuse grows:**

- Used on **one route** → `routes/<route>/-components/`
- Used on **several routes** → `routes/-components/` (e.g. landing sections) or `components/<name>/`
- Used **app-wide** → `components/<name>/`

Route files stay thin: `beforeLoad`, loader concerns, and composition. Heavy UI lives in `-components/`.

### `components/`

| Subfolder | Rule |
|-----------|------|
| `ui/` | **shadcn only.** Generated by codegen. Never add custom components here. |
| Everything else | Create a **named folder** that describes the concern (`navigation/`, `search/`, `creature-egg/`). Keep each folder small and readable — split when a folder becomes a junk drawer. |

Presentation-only. No fetchers, no business rules, no `queryOptions`.

### `data-access-layer/`

Defines **how** data is loaded. Nothing else.

**Put here:**

- `queryOptions` / `mutationOptions`
- TanStack DB **collection** definitions
- Shared **query key** prefixes (`query-keys.ts`)
- Generated or hand-rolled **typed API clients** (e.g. `data-access-layer/client/swagger-client.ts`), imported by domain files under `data-access-layer/<domain>/`

**Do not put here:**

- Custom hooks like `usePosts` (use `useQuery(postsQueryOptions)` in the component)
- UI components
- Unrelated business logic

**Fetching pattern — keep the chain flat:**

```typescript
// data-access-layer/posts/query-options.ts
async function fetchPosts() {
  return postsClient.list();
}

export const postsQueryOptions = queryOptions({
  queryKey: ["posts"],
  queryFn: fetchPosts,
});
```

```typescript
// routes/posts/index.tsx
const { data } = useSuspenseQuery(postsQueryOptions);
```

- If the fetch is **only used once** and trivial, **inline** it in `queryFn` — no separate fetcher file.
- If the fetch is **reused** (multiple query options, collections, server fn), keep the fetcher **in the same file**, above the `queryOptions` export.
- **Never** split into `posts-fetcher.ts` → `posts-query-options.ts` → `use-posts.ts` → component unless there is real reuse. That cyclical indirection makes tracing painful.

**Exception:** `data-access-layer/auth/viewer.ts` — auth/viewer session wiring may colocate middleware and viewer query options.

Server functions that mutate data often live in `modules/` and are **called from** `queryFn` / `mutationFn` here.

### `lib/`

Portable wrappers around **installed dependencies**. The litmus test: *could you copy this folder into another project's `lib/` and have it work once that dependency is installed?*

| Subfolder | Coupled to |
|-----------|------------|
| `tanstack/` | TanStack Form, Query, Router, Devtools |
| `db/` | Drizzle / database package |
| `envs/` | Environment parsing (Zod + `process.env` / `import.meta.env`) |
| `markdown/` | Shiki, markdown pipeline |
| `telegram/` | Telegram Bot API client wrapper |
| `utils.ts` | shadcn `cn()` (codegen) |

**Not for:** domain business logic, backstage rules, journal mapping, GitHub repo orchestration. That belongs in `modules/`.

### `modules/`

App-specific **business logic** that does not belong in `lib/` (not a dependency wrapper) and is not route-local.

Examples: `admin-auth/`, `backstage/`, `journal/`, `project-enrichment/`, `github/`.

Often contains TanStack Start `createServerFn` handlers, domain schemas, and mappers. Imported by `data-access-layer/` (for fetching) and occasionally by routes/workflows.

### `services/`

Thin **external HTTP/API** clients — one domain per folder (e.g. `services/openrouter/openrouter.api.ts`). Typed request/response functions only. No React, no query keys.

### `hooks/`

Global hooks for **UI behavior and local state** — debouncing, scroll motion, media queries, hotkeys, animation sequences.

**Do not add:**

- `usePosts`, `useUsers`, or any hook that only wraps `useQuery` / `useMutation`
- Data-fetching wrappers (use `queryOptions` + `useQuery` in the component)

**If a hook is truly route-specific**, put it in `routes/<route>/-hooks/` instead.

**Exception:** auth/viewer hooks may live in `data-access-layer/auth/` when they are part of session infrastructure.

### `utils/`

Standalone, **dependency-light helpers** (`slugify`, `errors`, `date-helpers`) that could be copied into another repo without dragging app context. Not for domain rules.

### `types/`

Global types that **cannot** be inferred from a typed client or server function return value.

- Prefer `ReturnType`, `Awaited<ReturnType<…>>`, and inference from `queryOptions` / server fns.
- Do **not** define a type here and then **cast** API responses to match it — fix the source typing instead.
- Few global types is healthy; many is usually a smell.

### `config/`

Static app metadata (`site.ts`, feature flags, link constants). No fetching.

### `data/`

Static fixtures and compile-time content (`data/portfolio/static.ts`). Not a substitute for `data-access-layer/`.

### `workflows/`

Multi-step async workflow definitions (e.g. project enrichment). Steps may call `modules/` and `lib/envs/`.

### `styles/`

CSS entry files imported from routes or root — not component-level Tailwind (that stays in components).

### Where does new code go?

```
Is it a route URL or route loader?
  → routes/

Is it UI only used on one route?
  → routes/<route>/-components/  (or -utils / -hooks)

Is it UI reused across routes?
  → routes/-components/  or  components/<concern>/

Is it shadcn primitives?
  → components/ui/  (codegen only)

Is it queryOptions, a collection, or a typed client?
  → data-access-layer/<domain>/

Is it coupled to an npm package (TanStack, Drizzle, Shiki, …)?
  → lib/<package>/

Is it app business logic (server fns, domain rules)?
  → modules/<domain>/

Is it a raw external API client?
  → services/<domain>/

Is it a generic string/date/url helper?
  → utils/

Is it a global type with no inference path?
  → types/  (sparingly)
```

## ✅ DO

### Routing

- ✅ Create routes in folders with `index.tsx` file
- ✅ Put route-specific components in `-components` folder
  ```
  routes/
    dashboard/
      index.tsx
      -components/
        DashboardCard.tsx
  ```
- ✅ Put reusable/global components in `src/components/`
- ✅ Use `beforeLoad` for auth and role checks
- ✅ Throw `redirect()` for unauthorized access

### React Query

- ✅ Create `queryOptions` and use hooks directly in components

  ```typescript
  export const usersQueryOptions = queryOptions({
    queryKey: ["users"],
    queryFn: async () => treatyClient.users.get(),
  });

  // In component
  const query = useSuspenseQuery(usersQueryOptions);
  ```

- ✅ Use `useSuspenseQuery` for required data
- ✅ Handle mutations inline in components
- ✅ Exception: Auth/viewer logic can have custom hooks

### Styling

- ✅ Use shadcn components for UI elements
- ✅ Use DaisyUI for:
  - Theme utilities: `bg-base-100`, `bg-base-200`, `text-base-content`
  - Button classes: `btn`, `btn-primary`, `btn-ghost`, `btn-sm`
  - Card classes: `card`, `card-body`
  - Other theme-related utilities
- ✅ Use `twMerge` for className merging
- ✅ Add `data-test` attributes for testability

### Forms

- ✅ Use TanStack Form with custom field components
- ✅ Validate with Zod inline
  ```typescript
  <form.AppField
    name="email"
    validators={{ onChange: z.string().email() }}>
    {(field) => <field.TextField label="Email" />}
  </form.AppField>
  ```

### Code Organization

- ✅ Use path alias `@/` for imports
- ✅ Import types from `@backend/` for end-to-end type safety
- ✅ Keep route logic in route files
- ✅ Extract reusable logic to custom hooks
- ✅ Use `satisfies` operator for type checking

## ❌ DON'T

### React 19 with Compiler

- ❌ DON'T use `useMemo` - React Compiler handles it
- ❌ DON'T use `useCallback` - React Compiler handles it
- ❌ DON'T manually optimize unless profiling shows issues

### React Query

- ❌ DON'T create custom hooks for every query
- ❌ DON'T wrap `queryOptions` in custom hooks (except auth/viewer)
- ❌ DON'T over-abstract data fetching

### Routing

- ❌ DON'T put components directly in route file if they're reusable
- ❌ DON'T create route files without folder structure
- ❌ DON'T skip `beforeLoad` for protected routes
- ❌ DON'T create flat route structure - use folders with index files

### Styling

- ❌ DON'T mix DaisyUI components with shadcn (stick to shadcn for UI)
- ❌ DON'T use DaisyUI except for theme utilities and button classes or simple component taht can no seroius accessibility require,nets an dis standalone enough
- ❌ DON'T forget responsive design (`md:`, `lg:` prefixes)
- ❌ DON'T hardcode colors - use theme variables

### File Organization

- ❌ DON'T put route-specific components in global `components/` folder
- ❌ DON'T create components in route file itself - use `-components` folder
- ❌ DON'T forget to use path aliases (`@/`)

## Quick Reference

### Route Structure

```typescript
// routes/dashboard/index.tsx
export const Route = createFileRoute("/dashboard")({
  component: DashboardPage,
  beforeLoad: async (context) => {
    if (!context.context.viewer?.user) {
      throw redirect({ to: "/auth", search: { returnTo: context.location.pathname } });
    }
  },
});

function DashboardPage() {
  const query = useSuspenseQuery(dataQueryOptions);
  return <div>...</div>;
}
```

### Query Pattern

```typescript
// Define queryOptions
export const itemsQueryOptions = queryOptions({
  queryKey: ["items"],
  queryFn: async () => {
    const { data, error } = await treatyClient.items.get();
    if (error) throw error;
    return data;
  },
});

// Use directly in component
const { data } = useSuspenseQuery(itemsQueryOptions);
```

### Mutation Pattern

```typescript
const mutation = useMutation({
  mutationFn: async (data) => treatyClient.items.post(data),
  onSuccess() {
    toast.success("Success!");
  },
  meta:{
    invalidates:[["items]]
  }
});
```

### Form Pattern

```typescript
const form = useAppForm({
  ...formOptions({ defaultValues: { name: "" } }),
  onSubmit: async ({ value }) => mutation.mutate(value),
});
```

### Styling Classes

```tsx
// DaisyUI theme utilities
<div className="bg-base-100 text-base-content">
  <button className="btn btn-primary">Action</button>
</div>

// shadcn components for everything else
<Card>
  <CardHeader>
    <CardTitle>Title</CardTitle>
  </CardHeader>
</Card>
```

## Common Mistakes to Avoid

1. Creating custom hooks for every React Query call
2. Using `useMemo`/`useCallback` with React 19 Compiler
3. Putting route components in wrong folders
4. Mixing DaisyUI components with shadcn (use shadcn for UI, DaisyUI for theme and simpler components that won't be nested in daisyui ones)
5. Forgetting `data-test` attributes
6. Not using `beforeLoad` for route protection
7. Creating routes without folder structure
8. Avoid casting to any or any types , request for permisson to do any such thig .Hiding type issues by casting t other pes is STRICTLY FORBIDDEN
9. Errors should be of type unknown in the catch block
